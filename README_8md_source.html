1 README
2 ======
3 
4 This is the Pathie project. It aims to provide a C++ library that covers
5 all needs of pathname manipulation and filename fiddling, without
6 having to worry about the underlying platform. That is, it is a glue
7 library that allows you to create platform-independent filename
8 handling code with special regard to Unicode path names.
9 
10 Supported systems
11 -----------------
12 
13 Currently supported platforms are Linux with GCC and Windows with GCC
14 (via MinGW). Any other compiler or system might or might not work.
15 
16 It is recommended to use at least GCC 4.9.0 due to full implementation
17 of C++11.
18 
19 The library
20 -----------
21 
22 The entire world is using UTF-8 as the primary Unicode encoding. The
23 entire world? No, a little company from Redmond resists the temptation
24 and instead uses UTF-16LE, causing cross-platform handling of Unicode
25 pathes to be a nightmare.
26 
27 One of the main problems the author ran into was compiler-dependant
28 code that was not marked as such. Many sites on the Internet claim
29 Unicode path handling on Windows is easy, but in fact, it only is if
30 you define “development for Windows” as “development with MSVC”,
31 Microsoft’s proprietary C/C++ compiler, which provides nonstandard
32 interfaces to allow for handling UTF-16LE filenames. The pathie
33 library has been developed with a focus on MinGW and crosscompilation
34 from Linux to Windows and thus does not suffer from this problem. It
35 is guaranteed to work both with native g++ on Linux, native g++ via
36 MinGW on Windows, and crosscompilation MinGW g++ on Linux. I will
37 accept patches for other compilers, but I will not thoroughly test on
38 them.
39 
40 Even more, Pathie’s author is of the strong opinion that UTF-8 is the
41 correct encoding to use, and is not willing to do differently on
42 Windows. Visit the [UTF8 Everywhere](http://www.utf8everywhere.org)
43 website if you need good arguments for this. However, the Windows
44 world of course disagrees, and this is where the pathie library comes
45 in. It allows you to do your entire pathname handling in UTF-8, and
46 transparently converts to whatever pathname encoding the underlying
47 system uses, such as UTF-16LE on Windows, without having you
48 notice.

50 Usage
51 -----
52 
53 First thing is to include the main header:
54 
55 ~~~~~~~~~~~~~~~~~~{.cpp}
56 #include <pathie.hpp>
57 ~~~~~~~~~~~~~~~~~~
58 
59 This will include any other headers as well. There should never be a
60 need to include pathie’s complement headers.
61 
62 Now consider the simple task to get all children of a directory, which
63 have Unicode filenames. Doing that manually will result in you having
64 to convert between UTF-8 and UTF-16 all the time. With pathie, you can
65 just do this:
66 
67 ~~~~~~~~~~~~~~~~~~~{.cpp}
68 std::vector<Pathie::Path> children = your_path.children();
69 ~~~~~~~~~~~~~~~~~~~
70 
71 Done. Retrieving the parent directory of your directory is easy as
72 hell:
73 
74 ~~~~~~~~~~~~~~~~~~~{.cpp}
75 Pathie::Path yourpath("foo/bar/baz");
76 Pathie::Path parent = yourpath.parent();
77 ~~~~~~~~~~~~~~~~~~~
78 
79 But Pathie is much more than just an abstraction of different filepath
80 encodings. It is a utility library for pathname manipulation, i.e. it
81 allows you to do things like finding the parent directory, resolving
82 symlinks (yes, including NTFS symlinks on Windows), expanding relative
83 to absolute pathes, decomposing a filename into basename, dirname, and
84 extension, and so on. See the documentation of the central
85 Pathie::Path class on what you can do.
86 
87 ~~~~~~~~~~~~~~~~~~~~~~{.cpp}
88 // Assume current directory is /tmp
89 Pathie::Path p("foo/bar/../baz");
90 p.expand(); // => /tmp/foo/baz
91 ~~~~~~~~~~~~~~~~~~~~~~
92 
93 Or my personal favourite:
94 
95 ~~~~~~~~~~~~~~~~~~~{.cpp}
96 Pathie::Path p1("/tmp/foo/bar");
97 Pathie::Path p2("/tmp/bar/foo");
98 Pathie::Path p3 = p1.relative(p2); // => ../../foo/bar
99 ~~~~~~~~~~~~~~~~~~~
100 
101 It also provides you with commonly used pathes like the user’s
102 configuration directory or the path to the running executable.
103 
104 ~~~~~~~~~~~~~~~~~~~~{.cpp}
105 Pathie::Path exepath = Pathie::Path::exe();
106 ~~~~~~~~~~~~~~~~~~~~
107 
108 And best of it? All arguments you pass to Pathie are UTF-8, and
109 everything you get back is encoded in UTF-8 as well.
110 
111 Still, if you interface directly with the Windows API or other external
112 libraries, you might want to retrieve the native representation from a
113 Path or construct a Path from the native representation. Pathie
114 doesn’t want to be in your way then:
115 
116 ~~~~~~~~~~~~~~~~~~~~{.cpp}
117 // Contruct from native
118 wchar_t* utf16 = Win32ApiCall();
119 Path mypath = Path::from_native(utf16); // also accepts std::wstring
120 
121 // Retrieve native (Note C++’ish std::wstring rather than
122 // raw wchar_t*)
123 std::wstring native_utf16 = mypath.native();
124 ~~~~~~~~~~~~~~~~~~~~
125 
126 On UNIX, these methods work with normal strings in the underlying
127 filesystem encoding. In most cases, that will be UTF-8, but some
128 legacy systems may still use something like ISO-8859-1 in which case
129 that will differ. Therefore, only use the `Path` constructor if you
130 are constructing from a UTF-8 string!
131 
132 ### Stream replacements
133 
134 Pathie mainly provides you with the means to handle pathes, compose,
135 and decompose them. Inspired by the [boost::nowide
136 library](http://cppcms.com/files/nowide/html/) library, however, it
137 also provides you with drop-in replacements for `std::ifstream` and
138 `std::ofstream` that flawlessly work on Unicode pathes even on
139 Windows. See the documentation of Pathie::ofstream for more
140 information.
141 
142 ~~~~~~~~~~~~~~~~~{.cpp}
143 Pathie::Path p("Bärenstark.txt");
144 Pathie::ofstream file(p);
145 file << "Some content" << std::endl;
146 file.close()
147 ~~~~~~~~~~~~~~~~~

149 Dependencies and linking
150 ------------------------
151 
152 Pathie is standalone, that is, it requires no other libraries except
153 for those provided by your operating system. Note that there’s a
154 caveat with this on Windows, which does provide the `shlwapi.lib`
155 library by default, but does not automatically link it in. Be sure to
156 add that library to your linking command when compiling for
157 Windows.
158 
159 It is recommended to link in pathie as a dynamic library, because
160 there are some problems with it when linked statically on certain
161 operating systems (see _Caveats_ below). If you are sure you aren’t
162 affected by those problems, it is possible to link in pathie
163 statically.
164 
165 Caveats
166 -------
167 
168 This library assumes that under all UNIX systems out there (I also
169 consider Mac OSX to be a UNIX system) the file system root always is
170 `/` and the directory separator also always is `/`. This structure is
171 mandatory as per POSIX -- in POSIX.1-2008, it’s specified in section
172 10.1. Systems which do neither follow POSIX directory structure, nor
173 are Windows, are unsupported.
174 
175 On POSIX-compliant systems other than Mac OS X, the filesystem
176 encoding [generally is
177 unspecified](https://unix.stackexchange.com/questions/2089/what-charset-encoding-is-used-for-filenames-and-paths-on-linux).
178 Pathnames are merely byte blobs which do not contain NUL bytes, and
179 components are separated by `/`. It’s up to the applications,
180 including utilities like a shell or the ls(1) program, to make
181 something of those byte streams. Therefore, it is perfectly possible
182 that on one system, user A uses ISO-8859-1 filenames and user B uses
183 UTF-8 filenames. Even the same user could use differently encoded
184 filenames. Programs that have to interpret the byte blobs in pathnames
185 on these systems look at the locale environment variables, namely
186 `LANG` and `LC_ALL`, see section 7 of POSIX.1-2008. As a consequence,
187 it may happen you want to create filenames with characters not
188 supported in the user’s pathname encoding. For example, if you want to
189 create a file with a hebrew filename and the user’s pathname encoding
190 is ISO-8859-1, there’s a problem, because ISO-8859-1 has no hebrew
191 characters in it, but in UTF-8, which is the encoding you are advised
192 to use and which is what Pathie’s API expects from you, they are
193 available. There is no sensible solution to this problem that the
194 Pathie library could dictate; the `iconv()` function used by pathie
195 just replaces characters that are unavailable in the target encoding
196 with a system-defined default (probably “?”). Note that on systems
197 which have a Unicode pathname encoding, especially modern Linuxes with
198 UTF-8, such a situation can’t ever arise, because the Unicode
199 encodings (UTF-*) cover all characters you can ever use.
200 
201 Note a possible problem in Pathie::Path::expand() when you link Pathie
202 in statically and depend on expanding pathes like `~username/foo` on
203 UNIX. See Pathie::Path::expand() docs for information on this.
204 
205 At least on FreeBSD, calling the POSIX `iconv()` function fails with
206 the cryptic error message “Service unavailable” if a program is linked
207 statically. I’ve reported [a bug on
208 this](https://bugs.freebsd.org/bugzilla/show_bug.cgi?id=196567). This
209 means that you currently can’t link in pathie statically on FreeBSD
210 and systems which don’t allow statically linked executables to call
211 `iconv()`.
212 
213 It is recommended to set your program’s locale to the environment’s
214 locale before you call any functions the Pathie library provides on a
215 UNIX system, because this will allow Pathie to use the correct
216 encoding for filenames on UNIX systems. This is relevant where the
217 environment’s encoding is not UTF-8, e.g. with $LANG set to
218 `de_DE.ISO-8859-1`. You can do this as follows (the `""` locale always
219 refers to the locale of the environment):
220 
221 ~~~~~~~~~~~~~~~~~~~~~{.cpp}
222 #include <locale>
223 std::locale::global(std::locale(""));
224 ~~~~~~~~~~~~~~~~~~~~~

226 Don’t you duplicate boost::filesystem?
227 -------------------------------------
228 
229 Yes and
230 no. [boost::filesystem](http://www.boost.org/doc/libs/1_56_0/libs/filesystem/doc/index.htm)
231 provides many methods pathie provides, but has a major problem with
232 Unicode path handling if you are not willing to do the UTF-8/UTF-16
233 conversion manually. boost::filesystem always uses UTF-8 to store the
234 pathes on UNIX, and, which is the problem, always uses UTF-16LE to
235 store the pathes on a Windows system. There is no way to override
236 this, although there is a [hidden documentation
237 page](http://www.boost.org/doc/libs/1_51_0/libs/locale/doc/html/default_encoding_under_windows.html)
238 that claims to solve the problem. I have wasted a great amount of time
239 to persuade boost::filesystem to automatically convert all
240 `std::string` input it receives into UTF-16LE, but failed to
241 succeed. Each time I wanted to create a file with a Unicode filename,
242 the test failed on Windows by producing garbage filenames. Finally I
243 found out that the neat trick shown in the documentation above indeed
244 does work -- but only if you use the Microsoft Visual C++ compiler
245 (MSVC) to compile your code. I don’t, I generally use g++ via the
246 [MinGW](http://www.mingw.org) toolchain. boost::filesystem fails with
247 g++ via MinGW with regard to Unicode filenames on Windows entirely and
248 completely as of this writing (September 2014).
249 
250 Apart from that, pathie provides some additional methods, especially
251 with regard to finding out where the user’s pathes are. It is modelled
252 after Ruby’s popular
253 [Pathname](http://ruby-doc.org/stdlib-2.1.2/libdoc/pathname/rdoc/Pathname.html#method-i-rmtree)
254 class, but it doesn’t entirely duplicate its interface (which wouldn’t
255 be idiomatic C++).
256 
257 Also, pathie is a small library. Adding it to your project shouldn’t
258 hurt too much, while boost::filesystem is quite a large dependency.
